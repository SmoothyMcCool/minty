Creating and Editing

Workflows are composed of individual tasks that perform specific actions, plus one special output task that runs once all other tasks have finished. The typical workflow begins with a PacketEmitter that carries the state needed to drive the rest of the pipeline, such as a query ID or other contextual data.

  • Tasks - The building blocks of a workflow. Connect them in the order that matches the business logic you need to implement.
  • PacketEmitter - The first task in most workflows. It emits a packet that contains the state information required by subsequent tasks.
  • Per-packet results - Some tasks keep a copy of the data they processed for later use. Refer to the individual task documentation for details on how to store, retrieve, or manipulate these results.
  • Array handling - Most tasks can operate on arrays of text or data. However, it is usually simpler to design the workflow so that each piece of information lives in its own packet. This avoids the need to split, normalize, duplicate, or combine arrays. The Task documentation provides guidance on how to perform these operations if you do need to work with arrays.

Output Tasks

Output tasks are special because they execute only once, after all other tasks have completed. They receive an ExecutionResult object that contains metadata and aggregated results from the entire workflow.

  • startTime - Timestamp when the workflow started.
  • endTime - Timestamp when the workflow finished.
  • logFile - Path or URL to the log file generated during execution.
  • results - A map keyed by step names (not task names). Each key maps to a list of JSON objects, one for every packet processed by that step. Example usage: assembling HTML fragments from multiple HTML Formatter tasks.
  • errors - Structured like results. Contains a list of error strings emitted by different tasks, if any.
